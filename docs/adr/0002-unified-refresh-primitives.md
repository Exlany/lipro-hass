# ADR-0002: 统一刷新与重同步原语

- **Status**: Accepted
- **Date**: 2026-03-12
- **Owners**: Coordinator / device lifecycle

## 背景

设备列表刷新、显式 refresh service、周期 refresh、MQTT 断连后的重同步，历史上容易被分散到多个 facade 或副作用入口。结果是：

- 调用方以为自己触发了“强制刷新”，实际只触发了 coordinator 的常规更新节拍
- 设备快照、订阅同步、监听器通知之间缺乏统一语义
- 刷新逻辑一旦分叉，性能优化与故障排查都会变难

## 决策

1. 统一使用 `Coordinator.async_refresh_devices()` 作为正式刷新入口
2. 统一使用 `Coordinator._async_refresh_device_snapshot()` 作为设备快照对齐原语
3. `DeviceRuntime` 负责设备快照生命周期；调用方不直接私连设备刷新细节
4. 显式 refresh、周期 refresh、状态重同步共享同一套刷新语义

## 取舍

### 收益

- “刷新”不再是模糊概念，而是可审计的正式原语
- 快照更新、MQTT 订阅同步、监听器更新可以保持单条时序
- 后续若要做缓存复用、批量优化或故障回退，只需守住一条主链

### 代价

- 一些历史 facade 需要退役，短期会带来文档与测试调整成本
- 调用方灵活性下降，但换来更强的一致性与可验证性

## 明确拒绝的替代方案

- 让不同 Service 各自实现“刷新”语义：会再次形成分叉
- 直接依赖 `async_request_refresh()` 代表强制设备刷新：语义不够明确
- 在 `DeviceRuntime` 外再保留增量刷新旁路：会放大索引一致性与状态偏移风险

## 后果

- 所有需要“重新对齐设备视图”的调用，都应围绕统一刷新原语建模
- 刷新性能问题优先在这条主链上解决，而不是再新增第二条刷新路径
- 如果未来需要更细粒度 refresh，也应作为统一原语的参数化扩展，而不是旁路实现

## 重议触发器

只有在以下情况之一出现时，才应重新讨论该决策：

1. 统一刷新原语成为明显性能瓶颈，且参数化扩展无法解决
2. 上游协议引入正式、可靠的差量订阅模型，可以安全替代当前快照主链
3. 设备规模扩大到必须拆成多级刷新域，而单条刷新原语已无法表达
