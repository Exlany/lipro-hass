# Phase 25.1 Context

**Phase:** `25.1 Snapshot atomicity and refresh arbitration`
**Milestone:** `v1.3 Quality-10 Remediation & Productization`
**Date:** `2026-03-17`
**Status:** `Ready for planning`
**Source:** `Phase 25` route freeze + runtime correctness review + current snapshot/device-runtime/coordinator behavior

## Why Phase 25.1 Exists

当前设备刷新主链仍允许“抓到一半 / 解析一半 / enrich 一半”的结果被当成新的全量 truth 发布到 coordinator state。对顶级质量标准而言，这不是“可接受降级”，而是**原子性缺失**。

## Goal

1. 让 full snapshot refresh 变成 all-or-nothing：fetch / parse / enrich 任一关键环节失败，都不得发布 partial truth。
2. 明确 refresh arbitration：拒绝提交、保留 last-known-good、抛出稳定可测的失败语义。
3. 让 coordinator 主链对这类失败给出正式、可测试的行为，而不是靠 debug log 猜测。

## Decisions (Locked)

- 本 phase 优先 correctness，不为兼容旧测试而保留 partial-success 语义。
- `DeviceRuntime` 允许保留 last-known-good snapshot，但不得把失败 refresh 伪装成成功 refresh。
- `Coordinator` 必须通过正式 update failure 路径暴露这类拒绝，而不是静默覆盖 state。
- malformed payload / non-dict rows / parse failure / mesh-group topology enrich failure 都属于 snapshot atomicity 范围，不再静默跳过。

## Non-Negotiable Constraints

- 不得引入第二条 refresh root；仍只能沿 `SnapshotBuilder -> DeviceRuntime -> Coordinator` 正式主链收口。
- 不得把“保留旧快照”实现成 silent success；失败必须沿主链可见、可测。
- 不得把 25.1 扩成 telemetry surface 或 trust-chain 主题；仅处理 runtime snapshot correctness。
