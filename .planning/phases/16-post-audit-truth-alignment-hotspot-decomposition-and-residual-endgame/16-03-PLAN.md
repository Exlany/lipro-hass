<objective>
把 control/service/config-flow 邻域的 auth / error / response contract 统一到更清晰的正式主链：`service_router.py` 继续只做 public handler home，`runtime_access.py` 与 diagnostics types 提供更稳定的 typed payload，`config_flow.py` 的登录/reauth 仲裁与 service path 使用同一套可判定语义。
</objective>

<tasks>
<task>收敛 `service_router.py`、`services/contracts.py`、`services/diagnostics/types.py` 中的 response shape，使 `send_command`、developer/share、diagnostics capability 与 runtime snapshot 不再依赖散落的 `dict[str, Any]` 约定。</task>
<task>收窄 `runtime_access.py` 与相邻 control seams 中的 `cast` / `getattr` / `import_module` 式探测，把 `RuntimeCoordinatorSnapshot`、entry/runtime lookup 与 telemetry projection 约束到更明确的 Protocol / alias / typed read-model。</task>
<task>统一 `services/command.py`、`services/errors.py`、diagnostics handlers 与 `service_router.py` 的 auth/error 主线，让服务失败在“认证失败 / 可选能力缺失 / 输入错误 / 运行异常”之间具备稳定、可测试的分类。</task>
<task>把 `config_flow.py` 中登录、reauth、reconfigure 的 catch-all 缩窄为 documented arbitration：输入校验、Lipro API 错误、malformed response 与 truly unexpected error 要使用同一套词汇与日志语义，而不是继续混成单一 `unknown` 胶水口。</task>
</tasks>

## Verification Criteria

- control/service/config-flow 关键入口共享同一套 auth/error vocabulary，调用者不再靠猜测或宽字典 shape 判断失败语义；
- `service_router.py` 热点下降但仍保留 public handler home 身份，没有把控制面重新拆成第二条 orchestration story；
- `runtime_access.py` 与 diagnostics response types 的 typing 更窄、更接近正式 contract，而不是继续用 `Any` / `cast` 掩盖边界漂移；
- `config_flow.py` 的 broad exception handling 被收紧或显式文档化，登录与 reauth 的异常语义对用户、日志与测试都可判定。

## Must-Haves

- `service_router.py` 必须继续是 public handler home
- `config_flow.py` 与 services path 必须共用同一 auth/error 主线
- response payload shape 要稳定、可类型化、可测试
- 不新增 control-plane backdoor、动态导入旁路或第二条 orchestration 链
