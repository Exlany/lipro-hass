# Plan 122-01: 收口 master audit ledger 并同步 v1.35 route truth

## Scope

- 为 `v1.35` 建立唯一可引用的 master audit ledger / table truth
- 把 ledger home、active route、3-plan 拆分与 requirements basket 同步进当前 planning selector family
- 保持 `v1.34` 仍是 latest archived baseline，而不是被新的 ledger 覆写

## Files

- `.planning/v1.35-MILESTONE-AUDIT.md`
- `.planning/PROJECT.md`
- `.planning/ROADMAP.md`
- `.planning/REQUIREMENTS.md`
- `.planning/STATE.md`
- `.planning/reviews/PROMOTED_PHASE_ASSETS.md`

## Implementation Steps

1. 在 `.planning/v1.35-MILESTONE-AUDIT.md` 建立单一 ledger，至少落表：问题来源、影响面、归类（first-hop / appendix / metadata / guard / route truth）、处置状态、对应证据。
2. 把最新审阅结论与当前真实改动切片并入 ledger，明确区分：已落地、Phase 122 待收口、repo-external honest blocker、明确 deferred。
3. 同步 `.planning/PROJECT.md`、`.planning/ROADMAP.md`、`.planning/REQUIREMENTS.md`、`.planning/STATE.md`，确保它们共同承认同一 ledger home、同一 active route 文案与同一 3-plan 结构。
4. 若 ledger 需要成为长期治理/CI 证据，则补 `.planning/reviews/PROMOTED_PHASE_ASSETS.md`；若不提升，也要显式写明它不是 phase folklore 的替身。

## Validation

- `uv run pytest tests/meta/test_governance_guards.py tests/meta/test_governance_route_handoff_smoke.py tests/meta/test_governance_followup_route.py tests/meta/governance_current_truth.py tests/meta/governance_followup_route_current_milestones.py -q`
- `uv run pytest tests/meta/test_governance_promoted_phase_assets.py tests/meta/test_governance_release_continuity.py tests/meta/test_governance_release_docs.py -q`
- `uv run python scripts/check_file_matrix.py --check`

## Risks

- ledger 若只是复制 `v1.34` archived audit，而不是明确承接 `v1.35` active-route 残留，会形成第二套故事
- 若 route docs 只同步部分文件，`PROJECT / ROADMAP / REQUIREMENTS / STATE` 很快又会分叉
- 若 promoted / non-promoted 身份不明确，后续 review 会再次把 phase 资产误当治理真源

## Done When

- `v1.35` 存在唯一可引用的 master audit ledger，且每个剩余项都可追到 source / owner / status / evidence
- `PROJECT / ROADMAP / REQUIREMENTS / STATE` 对 `v1.35 active milestone route / starting from latest archived baseline = v1.34` 与 ledger home 的表述完全一致
- `v1.34` archived pointer 保持不变，ledger 的 promoted 身份也已明确
