# Phase 7.4 Research

**Phase:** `7.4 Protocol Replay / Simulator Harness 建立`
**Source milestone research:** `.planning/research/SUMMARY.md`
**Date:** 2026-03-13

## Findings

### 1. 正确的 replay truth chain

`07.4` 的目标不是做第二套 protocol/runtime 实现，而是把：

`authority fixtures + replay manifests -> deterministic driver -> LiproProtocolFacade / boundary decoders -> canonical assertions + drift assertions + telemetry assertions`

收敛成单一可回放证据链。

这意味着：
- replay harness 必须复用正式 `LiproProtocolFacade`、`core/protocol/boundary/*` 与 canonical contracts；
- replay driver 只能做装载、排序、时钟/seed 控制与断言编排；
- telemetry assertions 必须读取 `07.3 RuntimeTelemetryExporter` 的正式输出，不得自造第二套 observability truth；
- replay 结论必须可追溯到 authority source，而不是只依赖测试私有构造。

### 2. 已有 authority / fixture 基座应复用，而不是复制

仓库已经有可直接复用的 replay-ready 资产：

- `tests/fixtures/api_contracts/*`
  - 高漂移 REST 边界合同样本；
- `tests/fixtures/protocol_boundary/*`
  - decoder family 的 replay-ready 边界样本；
- `tests/fixtures/protocol_boundary/README.md`
  - 已明确“引用 authority source，而非复制第二真源”；
- `.planning/phases/02.6-external-boundary-convergence/02.6-BOUNDARY-INVENTORY.md`
  - external boundary authority / fixture / consumer inventory；
- `.planning/baseline/AUTHORITY_MATRIX.md`
  - 当前正式 authority source 仲裁点。

因此 `07.4` 最优解不是新造一套孤立 fixture 仓库，而是新增 `tests/fixtures/protocol_replay/` 作为 **manifest/index family**：
- 指向既有 authority fixture；
- 记录 replay scenario、版本戳、seed、时间基准、预期 assertions；
- 避免把 `api_contracts` 与 `protocol_boundary` 内容复制一份。

### 3. deterministic controls 应是 harness 的正式职责

对于 replay harness，最关键的不是“功能能跑”，而是：
- 固定样本顺序；
- 固定时间基准 / seed；
- 固定 error classification；
- 固定 REST/MQTT 输入到 canonical output 的比较方式。

因此最佳落点是 assurance-only home：
- `tests/harness/protocol/`：driver / loader / assertion helpers / replay report；
- `tests/fixtures/protocol_replay/`：scenario manifests；
- `tests/integration/` 与 `tests/meta/`：黑盒与 authority/drift 守卫。

不应把 deterministic controls 塞进 production `custom_components/` 主链。

### 4. telemetry assertions 的正确接法

`07.4` 的 telemetry assertions 只能建立在 `07.3` 已正式化的 exporter truth 上：
- replay scenario 执行后，通过正式 exporter / control telemetry surface 读取 snapshot/view；
- 对比 `auth recovery`、`MQTT reconnect`、`command confirmation latency`、`refresh latency` 等预期信号；
- 使用 `entry_ref` / `device_ref` 这类报告内稳定、跨报告不可关联的伪匿名引用；
- 保留真实时间戳以对齐 replay 序列与 AI 分析；
- 禁止 token/secret/`password_hash`/raw payload 进入 replay telemetry assertions。

### 5. `07.4` 的最佳技术姿态

本 phase 仍应保持“先进但克制”：
- 使用 `pytest`、现有 fixture family、轻量 deterministic driver；
- 不引入 broker/cloud 全仿真平台；
- 不引入外部 replay 服务、重型 tracing stack 或第二 protocol adapter；
- 若需要 artifact/report，优先在 `tests/harness/protocol/` 生成轻量 run summary，而不是把 replay 做成 runtime 子系统。

### 6. 与 `07.3 / 07.5 / 08` 的清晰边界

- `07.3` 拥有 telemetry contracts / redaction / cardinality / exporter truth；
- `07.4` 拥有 replay corpus / deterministic driver / replay assertions / replay run summary；
- `07.5` 拥有 governance closeout、matrix 同步、delete gate 与 verification package；
- `08` 拥有 AI Debug Evidence Pack 的统一打包导出。

`07.4` 不应越权：
- 改写 telemetry schema；
- 重写 boundary authority；
- 提前承担 evidence pack exporter 责任。

## Risks To Avoid

- replay 复制第二套 `REST/MQTT` 解码或 protocol root；
- 为了方便，直接消费 raw payload 或 coordinator internals；
- 为 replay 再造第二套 telemetry counters / report schema；
- 把 replay manifests 与 authority fixtures 双写成两套长期分叉的真源；
- 让 deterministic controls 依赖 wall-clock 或随机顺序，导致回放结果不可重复。

## Validation Architecture

`07.4` 的验证必须同时证明：
1. replay harness 只走正式 façade / decoder path；
2. replay corpus 具有 authority source、版本戳、deterministic controls；
3. REST/MQTT replays 能产出 canonical / drift / telemetry assertions；
4. replay run summary 能作为 `07.5` closeout 与 `08` evidence pack 的正式上游输入。

## Final Arbitration

`07.4` 的最优实现不是“把逆向协议再写一遍”，而是：
- 用 authority-indexed manifests + deterministic driver 复用正式 protocol public path；
- 把 replay 结果固化为 canonical / drift / telemetry 三类证据；
- 为 `07.5` 的 closeout 与 `08` 的 AI evidence pack 提供稳定、可追溯、可机器分析的上游输出。
