# Phase 7.3 Context

**Phase:** `7.3 Runtime Telemetry Exporter 正式化`
**Milestone:** `v1.1 Protocol Fidelity & Operability`
**Date:** 2026-03-13
**Status:** Ready for planning
**Source:** Roadmap + Requirements + Phase 7.2 closeout + full-repo audit

## User / Product Context

- `Phase 7.2` 已完成 architecture enforcement，后续 runtime telemetry 不能另起第二条主链，必须成为北极星正式主链中的 observer/export surface。
- 用户明确要求把全仓审查结论吸收到后续计划中，尤其是：compat 收口、超大文件拆薄、显式 contract 替代动态代理、黑盒主链验证增强、安全敏感资产收紧。
- 本 phase 不是做“日志堆叠”，而是把 runtime/protocol 的 operability signals 提炼成统一 telemetry truth，服务 diagnostics、system health、开发者定位与后续 replay/assertion。

## Audit-Driven Inputs

- 全仓评价为 `A- / 8.6`，方向正确但尚未达到终态最佳；最主要架构债仍是 `compat` 残留与少量超大文件。
- `custom_components/lipro/core/api/client.py`、`custom_components/lipro/core/coordinator/coordinator.py`、`custom_components/lipro/services/wiring.py` 已被识别为高复杂度热点；本 phase 需要避免再把 telemetry 逻辑堆进这些热点文件。
- `LiproClient`、`__getattr__`、legacy wiring re-export 等兼容壳层仍存在；telemetry surface 不能建立在 compat shell 上，而必须从正式 `protocol root` / `runtime root` 抽取。
- 测试体系很强，但偏白盒；本 phase 计划必须显式为 `config_flow -> setup_entry -> coordinator -> mqtt/service/diagnostics` 主链增加更厚黑盒集成验证铺垫。
- `password_hash` 应被视为高敏感资产；telemetry / diagnostics / system-health 的 redaction 必须把它当作凭证等价物。
- 个别 capability 语义仍有轻微漂移风险；telemetry schema 不能扩大字符串散落，必须以稳定 event/field contracts 收口。

## Architectural Context

- `LiproProtocolFacade` 仍是唯一正式 protocol root；它可以提供 protocol telemetry source，但不能因 telemetry 而退化为日志聚合器。
- `Coordinator` 仍是唯一正式 runtime root；runtime telemetry 必须从正式 lifecycle / command / refresh / mqtt orchestration seam 产生。
- `diagnostics.py`、`system_health.py` 仍必须保持 thin adapter；telemetry truth 不能在这些 HA adapter 内二次定义。
- `RuntimeAccess` 与 control-plane surfaces 仍是 diagnostics/system health 触达 runtime state 的正式路径。
- `07.2` 已为 `observer-only components` 预留 extension hook；`07.3` 必须使用该 hook，而不是回退为新的 side-root。

## Phase Goal

把 runtime/protocol telemetry 收口到单一 exporter surface：

1. 形成稳定的 telemetry event/model/schema/contracts；
2. 为 diagnostics、system health、developer/CI sinks 提供统一 truth；
3. 把 redaction、cardinality、retention、sampling 规则制度化；
4. 为 `07.4 replay harness` 提供可复用 telemetry assertions，而不复制第二套实现。

## Decision Updates (North Star 2.0 + AI Debug)

根据契约者裁决与北极星 2.0（HA-only）原则，本 phase 进一步锁定以下实现裁决：

- **AI 分析友好**：exporter 的 sinks 必须输出结构化、稳定字段集合，便于后续 AI 辅助定位与回放分析。
- **伪匿名化（推荐裁决）**：允许输出 *报告内稳定* 的伪匿名标识（例如 `device_ref`/`entry_ref`），使用伪装字符或派生短指纹；但必须满足：
  - 不能反推出真实 device_id/phone/user_id；
  - 不能跨报告长期关联（避免形成隐私追踪键）；
  - token/secret/`password_hash` 等凭证等价物永不出现在任何 sink。
- **Exporter 以 pull 为主**：exporter 从 protocol/runtime sources 拉取快照与“有界事件摘要”。sources 可维护 ring-buffer，但 exporter 本身不作为事件总线。
- **允许真实时间戳**：为满足 AI 调试与问题定位，本 phase 允许在 telemetry sinks 中包含真实时间戳字段（但仍须遵守脱敏与基数预算）。

## Non-Negotiable Constraints

- 不得把 exporter 设计成新 runtime root、事件总线中枢或另一个 service locator。
- 不得让 diagnostics/system health 直接拼接 coordinator/protocol internals 来构造 telemetry。
- 不得把 raw payload、敏感字段或未脱敏凭证样本写入 telemetry truth。
- 不得为了“先进”引入与 HA 场景不匹配的重型 observability stack。
- 必须维持 `uv run`、`pytest`、`ruff`、`mypy` 与现有 meta guards 的工程节奏。

## Planning Expectations

- `07.3-01` 应优先建立 telemetry exporter 的 contracts / ports / sink taxonomy / ownership，并把审查报告中的复杂度热点隔离在正式协作者层之外。
- `07.3-02` 应把 auth recovery、MQTT reconnect、command confirmation、refresh latency 等 source 接到 exporter，并补强 redaction 与黑盒主链验证。
- plans 必须显式写出：新增/重构哪些文件、如何避免继续膨胀 `client.py` / `coordinator.py` / `wiring.py`、如何与 `07.4/07.5` 对接。
- 如审查结论中存在不应在 `07.3` 内直接实现的事项，必须写入 deferred / forward hooks，而不是丢失。

## Deferred But Relevant

- compat shell 的最终清退仍需跨 phase 推进；`07.3` 只允许切断其对 telemetry truth 的依赖，不在本 phase 一次性做全仓删除。
- 更厚的黑盒主链验证可在 `07.3` 起步并在 `07.4/07.5` 继续深化。
- `password_hash` 的全链路高敏资产治理若涉及 broader persistence/UX policy，可在 `07.5` 再做统一 closeout。
