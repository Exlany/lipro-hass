# Phase 7.3 Research

**Phase:** `7.3 Runtime Telemetry Exporter 正式化`
**Source milestone research:** `.planning/research/SUMMARY.md`
**Date:** 2026-03-13

## Findings

### 1. 正确的 telemetry truth chain

`07.3` 的目标不是新增一套“更会记日志”的子系统，而是把：

`ProtocolTelemetry + ProtocolDiagnosticsContext + CoordinatorTelemetryService -> RuntimeTelemetryExporter -> diagnostics / system_health / developer / CI sinks`

收敛成单一真相链。

这意味着：
- `LiproProtocolFacade` 之下已有的 `ProtocolTelemetry` 继续作为 **protocol-plane source**；
- `CoordinatorTelemetryService` 继续作为 **runtime-plane source**；
- 新的 exporter 只能是 **observer / projection root**，不能参与编排、刷新、鉴权或命令决策；
- `diagnostics.py` / `system_health.py` 只能消费 exporter 投影，不得各自重新拼 telemetry 事实。

### 2. 当前 telemetry 种子与最优归位

当前仓库已经存在可复用的 telemetry 基座：

- `custom_components/lipro/core/protocol/telemetry.py`
  - 负责 MQTT facade / start / stop / sync / error 等 protocol summary；
- `custom_components/lipro/core/protocol/diagnostics_context.py`
  - 持有 protocol session + telemetry 的稳定 diagnostics snapshot；
- `custom_components/lipro/core/coordinator/services/telemetry_service.py`
  - 已能产出 `mqtt / command / status / tuning / signals` 的 runtime snapshot；
- `custom_components/lipro/control/runtime_access.py`
  - 当前承担 control-plane runtime snapshot 建构，但仍包含 `getattr` 式宽口读取；
- `custom_components/lipro/control/redaction.py`
  - 已拥有较成熟的脱敏规则，其中 `password_hash` 已在 redaction 集合中。

最优裁决不是重写这些种子，而是：
- 把它们收口为 exporter 的 **显式 sources**；
- 用新的 typed contracts / ports 把 `getattr` / 动态代理 / compat 依赖切断；
- 让 `07.3` 成为 telemetry 正式导出面，而不是再长一个“诊断专用实现”。

### 3. 07.3 应正式化的结构

本 phase 最值得正式化的结构家族如下：

- `Telemetry contracts`
  - event kind / snapshot sections / sink view / redaction classification / cardinality budget；
- `Telemetry ports`
  - `ProtocolTelemetrySource`、`RuntimeTelemetrySource`、`TelemetrySink` 等显式协定；
- `Exporter root`
  - 统一收集 protocol/runtime sources，生成稳定 exporter snapshot 与 sink projections；
- `Sink adapters`
  - diagnostics sink、system-health sink、developer/CI sink；
- `Governance truth`
  - `PUBLIC_SURFACES / VERIFICATION_MATRIX / FILE_MATRIX / RESIDUAL_LEDGER` 必须登记 exporter 的 owner、边界与 delete gate。

### 4. 审查报告的映射裁决

全仓审查报告中的关键结论，应在 `07.3` 中这样吸收：

- **compat 残留仍明显**
  - `07.3` 不直接负责全量删除 compat；
  - 但必须确保 exporter 不依赖 `LiproClient`、`__getattr__` 代理或 legacy wiring re-export。
- **`client.py` / `coordinator.py` / `wiring.py` 偏大**
  - `07.3` 不得把 exporter 逻辑继续塞回这些热点；
  - 应通过新的 `core/telemetry/*` 与窄协作者接口承接复杂度。
- **局部类型系统脆弱**
  - `07.3` 必须优先用 `Protocol`、`TypedDict`、`dataclass(slots=True)` 与显式 adapter，减少 `getattr` / duck typing。
- **测试偏白盒**
  - `07.3` 必须增加更厚的黑盒主链验证：`config_flow -> setup_entry -> runtime -> diagnostics/system_health/export`；
  - 单元测试仍保留，但不能是唯一证据。
- **`password_hash` 是凭证等价物**
  - exporter contracts、sink projections、diagnostics/system-health tests 都必须把它当高敏感资产处理；
  - 不仅要 redact，还要防止进入 label/cardinality-heavy 字段。

### 5. 最优技术立场

`07.3` 仍应保持北极星的“先进但克制”技术立场：

- 保持 `TypedDict + dataclass(slots=True) + Protocol` 为主；
- 保持 `uv + pytest + ruff + mypy` 为工程节奏；
- 不引入 `Prometheus` / `OpenTelemetry` / 事件总线 / DI 容器；
- 不把 exporter 做成独立 runtime daemon；
- 若需要 request-level timing，优先轻量 hook（如现有 request policy / trace seam），而不是重型基础设施。

### 6. 与 07.4 / 07.5 的衔接

- `07.4 replay harness` 必须复用 exporter 输出做 telemetry assertions；
- `07.5 governance/verification closeout` 必须登记 exporter family、sink owner、残留 delete gate 与验收命令；
- `07.3` 应为后续 phase 预留：
  - replay sink / replay assertion view
  - CI artifact sink
  - security closeout hooks（特别是 `password_hash` 等高敏资产治理）

## Recommended Outputs

- `custom_components/lipro/core/telemetry/models.py`
- `custom_components/lipro/core/telemetry/ports.py`
- `custom_components/lipro/core/telemetry/exporter.py`
- `custom_components/lipro/core/telemetry/sinks.py`

建议重构：
- `custom_components/lipro/core/coordinator/services/telemetry_service.py`
- `custom_components/lipro/core/protocol/telemetry.py`
- `custom_components/lipro/control/runtime_access.py`
- `custom_components/lipro/diagnostics.py`
- `custom_components/lipro/system_health.py`
- `custom_components/lipro/control/redaction.py`

原则：
- runtime / protocol 继续拥有 source truth；
- exporter 只做 normalize / project / redact / budget；
- sinks 只做目标视图投影，不反向建 truth。

## Risks To Avoid

- 把 exporter 做成新的 runtime root、service locator 或事件总线；
- diagnostics / system health / exporter 三套事实并存；
- 继续让 `control/runtime_access.py` 通过 `getattr` 漫游 coordinator internals；
- 让 `client.py` / `coordinator.py` / `wiring.py` 再次吸收 exporter 复杂度；
- 把 `password_hash`、device id、phone、token、raw error payload 放进 exporter 真相或高 cardinality 字段；
- 为了“更先进”而引入与 HA 插件不匹配的外部监控栈。

## Validation Architecture

`07.3` 的验证必须同时证明：
1. exporter 是 observer-only surface，而不是新 root；
2. diagnostics / system health / developer/CI sinks 共用单一 telemetry truth；
3. 红线字段（尤其 `password_hash`）不会泄露到 exporter 视图；
4. 黑盒主链能够从 HA 入口走到 exporter sink，而不依赖白盒 patch 才成立。

## Proposed Verification

- exporter unit tests 验证 source merge、sink projection、cardinality budget、sensitivity handling；
- diagnostics / system health tests 验证它们只消费 exporter 视图，不再二次定义 telemetry truth；
- 集成测试验证 `setup_entry -> runtime_data -> telemetry exporter -> diagnostics/system_health` 主链；
- governance docs 验证 exporter family 与 sinks 已在矩阵中登记。

## Final Arbitration

`07.3` 的最优实现不是“做一个更炫的 observability 框架”，而是：
- 用显式 contracts/ports 把已有 runtime/protocol telemetry 种子正式化；
- 让 exporter 成为单一投影真源；
- 吸收审查报告中的 compat、复杂度、安全、测试问题，但不把本 phase 变成无边界的大清仓。
