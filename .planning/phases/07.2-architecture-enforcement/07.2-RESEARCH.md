# Phase 7.2 Research

**Phase:** `7.2 Architecture Enforcement 加固`
**Source milestone research:** `.planning/research/SUMMARY.md`
**Date:** 2026-03-13

## Findings

### 1. 正确的 enforcement 主链

`07.2` 的目标不是继续堆更多零散 denylist，而是把：

`North Star / baseline docs -> architecture policy spec -> shared policy helpers -> local-fast script + focused meta guards + CI gate`

收敛成一条单一裁决链。

这意味着：
- 北极星与 baseline 仍是正式真源；
- policy spec 只是把 baseline 翻译成可执行规则，而不是第二权威；
- tests/scripts/CI 只是执行与仲裁层，不反向定义架构真相。

### 2. 07.2 应固化的 rule taxonomy

本 phase 最值得正式化的规则家族如下：

- `Plane policy`
  - 五平面依赖方向；
  - `Runtime` 只消费 `LiproProtocolFacade + canonical contracts`；
  - `Control` 只消费 runtime public surface；
  - `Assurance` 只观察与仲裁，不拿业务编排权。
- `Root policy`
  - `LiproProtocolFacade` 是唯一正式 protocol root；
  - `Coordinator` 是唯一正式 runtime root；
  - 禁止 split-root、child façade 回升、旁路 root 再合法化。
- `Surface policy`
  - public surface 必须明确区分 `canonical / transitional / forbidden`；
  - `LiproClient`、`LiproMqttClient` 只能停留在 transitional compat shell。
- `Boundary-local policy`
  - `core/protocol/boundary/*` 是 protocol-plane 内部 collaborator family；
  - 不能被 runtime / control / domain / entity / platform 当 public surface 消费。
- `Raw-payload policy`
  - raw vendor payload 不得穿透 protocol boundary；
  - runtime / domain / entity / control 只吃 canonical contracts。
- `Compat / residual policy`
  - compat 只能显式、可计数、可删除；
  - 例外必须回写 residual ledger / delete gate，而不能散落在测试 allowlist。
- `Authority policy`
  - `North Star -> baseline docs -> reviews -> phase docs/tests/code evidence` 的顺序不能反转；
  - 所有规则都要能回答“源自哪份真源、由谁维护、怎么验证”。
- `Backdoor / bypass policy`
  - 禁止 runtime internals backdoor、私有 auth seam、边界泄漏与 coordinator/property 漂移。

### 3. 最优技术立场

`07.2` 不需要为了“更先进”而引入更重的规则引擎；当前最优技术栈仍是：

- `Python 3.13`
- `pytest` + AST/meta guards
- `uv` 统一执行
- 共享 policy helper + fail-fast script
- 现有 GitHub Actions governance job 继续承载架构门禁

不推荐在本 phase 起点就引入：
- `import-linter/grimp`
- 独立 policy DSL / shadow config tree
- changed-files 魔法或过度复杂的本地快检
- 新的运行时依赖

### 4. 07.2 的核心架构裁决

#### 4.1 单一 truth chain

本织师裁决：

- **正式 human-readable policy truth** 应落在：
  - `.planning/baseline/ARCHITECTURE_POLICY.md`
- **可执行投影** 应落在：
  - `tests/helpers/architecture_policy.py`
  - `tests/helpers/ast_guard_utils.py`
  - `scripts/check_architecture_policy.py`

其中：
- `ARCHITECTURE_POLICY.md` 是 baseline 真源的一部分；
- helper/script 从 baseline docs 派生规则并执行；
- 不再引入手工维护的第二份 YAML/JSON 真源。

#### 4.2 为什么不选手工维护 YAML

虽然 machine-readable YAML 看起来更“现代”，但在当前北极星约束下，它很容易变成第二真源。

`07.2` 更优雅的做法是：
- baseline 继续由 Markdown 资产承载；
- 通过稳定 rule table / rule id / source ref 让 helper/script 可解析；
- 如未来复杂度继续升高，再评估“由 baseline 自动生成 machine projection”，而不是现在就手工双写两份权威资产。

### 5. 推荐输出

#### 5.1 Policy baseline

新增：
- `.planning/baseline/ARCHITECTURE_POLICY.md`

建议每条规则最少包含：
- `rule_id`
- `taxonomy`
- `source_docs`
- `governed_paths / consumers`
- `allowed / forbidden`
- `enforcement_kind`
- `residual_gate`
- `future_extension`

#### 5.2 Executable helpers

新增：
- `tests/helpers/architecture_policy.py`
- `tests/helpers/ast_guard_utils.py`
- `scripts/check_architecture_policy.py`

重构：
- `tests/meta/test_dependency_guards.py`
- `tests/meta/test_public_surface_guards.py`
- `tests/meta/test_governance_guards.py`
- `scripts/check_file_matrix.py`

原则：
- `check_file_matrix.py` 继续只管 inventory / governance drift；
- `check_architecture_policy.py` 专管 plane/root/surface/authority enforcement；
- meta tests 变成 shared policy 的 focused regression shells。

### 6. Local-fast + CI-fail-fast 最优组合

#### Local-fast

推荐最小正式链路：
- `uv run python scripts/check_architecture_policy.py --check`
- `uv run python scripts/check_file_matrix.py --check`
- `uv run pytest -q -x tests/meta/test_dependency_guards.py tests/meta/test_public_surface_guards.py tests/meta/test_governance_guards.py`

特点：
- 全仓扫描但只限轻量结构守卫；
- 不做 changed-files 差分魔法；
- 比功能测试更早、更轻、更明确地阻断架构偏航。

#### CI-fail-fast

推荐复用现有 `governance` job，并把顺序升级为：
1. `check_architecture_policy.py`
2. `check_file_matrix.py --check`
3. `tests/meta/test_dependency_guards.py`
4. `tests/meta/test_public_surface_guards.py`
5. `tests/meta/test_governance_guards.py`

### 7. 对 07.3 / 07.4 的前向兼容

- `07.3 telemetry exporter` 必须被建模为 observer/export surface，而不是新 runtime root；
- `07.4 replay harness` 必须被建模为 assurance-only consumer，而不是第二 protocol implementation；
- 因此 `07.2` 的 policy taxonomy 要预留两类 extension slot：
  - `observer-only components`
  - `assurance-only boundary consumers`

这样后续 phase 只需登记新 family / new governed paths，而不需要重写 enforcement 哲学。

## Required Outputs

- `.planning/baseline/ARCHITECTURE_POLICY.md` 的目标设计
- shared policy helper / AST helper / fail-fast script 的正式落点
- meta guards 与 governance CI 的升级顺序
- `07.3 / 07.4` 的 extension hooks

## Risks To Avoid

- 把 `ARCHITECTURE_POLICY` 做成第二真源
- 继续扩散手写 point-fix denylist，而不收敛 shared policy
- 把 `check_file_matrix.py` 演化成 mega-checker
- 让 `07.2` 越权提前冻结 `07.3 telemetry schema` 或 `07.4 replay API`
- 为“更先进”而过早引入重型规则引擎，反而增加理解成本
