# Phase 02.6 Research: External Boundary Convergence

**Updated:** 2026-03-12
**Research mode:** boundary formalization / authority arbitration
**Decision stance:** 不受历史成本约束，以“边界必须有 formal owner + authority + fixtures + drift rules”为唯一判断标准

## Research Framing

本 research 不回答“现有测试够不够多”，而回答：

- 哪些外部边界已经真实存在，却还没有正式 owner / authority 资产？
- `docs / fixtures / generated / implementation` 的同步方向应该怎样锁定，才不会继续由代码暗中定义真相？
- 哪些边界可以复用 Phase 1 / 2.5 的既有真源，哪些必须在 2.6 新增 asset family？
- 怎样把 2.6 设计成 Phase 3 的真实前置，而不是一份笼统的“以后补文档”清单？

结论先行：

- **当前最薄弱的不是行为测试数量，而是 authority-source 与 generated-contract 缺位。**
- **Firmware 的核心不是“再多抓一个远端样例”，而是先写死 `local trust root > remote advisory > implementation adapter`。**
- **Share worker / developer feedback / support payload 必须被当作 generated external contracts 来治理，而不是仅当成 service helper 输出。**
- **`get_city / query_user_cloud` 已经有强 protocol fixture，不应在 2.6 被重复定义；正确做法是纳入 external-boundary inventory。**

## Boundary Families

### 1. Share Worker Boundary

包含：
- anonymous share upload
- install token refresh
- developer feedback submit
- lite payload fallback / retry / backoff

**Why it matters**：
- 它已经是明确的出站 boundary，有远端 host、有 body 语义、有 retry/backoff 规则
- 当前已有大量行为测试，但几乎没有 golden fixture / body snapshot / authority doc

**Current owners in code**：
- payload build: `core/anonymous_share/report_builder.py`
- submit/refresh transport: `core/anonymous_share/share_client.py`
- workflow ownership: `core/anonymous_share/manager.py`
- HA service surface: `services/share.py`、`services/diagnostics/helpers.py`

### 2. Firmware Advisory Boundary

包含：
- bundled `firmware_support_manifest.json`
- remote `firmware-support` advisory endpoints
- OTA rows v1/v2/controller external responses

**Why it matters**：
- 这里同时存在 local bundled artifact、remote advisory source、runtime consumer 三层语义
- 若 authority 不锁死，后续最容易出现“远端数据偷偷放宽 certified”或“本地 manifest 与 parser 双口径”

**Locked interpretation**：
- local bundled JSON = trust root
- remote firmware-support endpoints = advisory only
- implementation adapter / candidate resolution = consumer logic

### 3. Support Payload Boundary

包含：
- developer feedback payload
- diagnostics support payload
- developer report / debug snapshots 中用于外发的 generated reports

**Why it matters**：
- 这是 Phase 3 diagnostics/support surface 的直接上游
- 当前最薄弱点是 `generated` 没有 authority source，也没有动态字段归一化规则

### 4. Diagnostics Capability Boundary

包含：
- `get_city`
- `query_user_cloud`
- `query_command_result`
- body/door sensor history 类 optional capability endpoints

**Why it matters**：
- 它们属于 external endpoint family，但 strength 不同：`get_city / query_user_cloud` 已有 Phase 1 fixture truth，其他 still behavior-only
- 2.6 应把它们统一纳入 inventory，并按强/弱 contract 面分层治理

## Cross-Phase Arbitration

- **Phase 1** 继续拥有 `get_city / query_user_cloud` 的 canonical protocol truth；Phase 2.6 只登记并延用，不重定义。
- **Phase 2 / 2.5** 继续拥有 protocol root 与 normalized contract 归属；Phase 2.6 不得把 external-boundary owner 写成 split-root folklore。
- **Phase 3** 必须消费 2.6 的 outputs，而不是在 diagnostics/support surface 内再次定义 authority source。
- **Phase 5** 才统一 runtime plane 的正式 public surface；2.6 只 formalize external boundaries，不介入 runtime ownership。
- **Phase 6** 再把 2.6 的 authority/fixture/drift rules 上收为 meta guards 与 CI audits。

## Key Findings

### 1. `generated` 是当前最大空洞

`AUTHORITY_MATRIX` 写了 `generated`，却没有 `generated` 真源行；这意味着：
- anonymous share payload
- developer feedback payload
- diagnostics/support generated reports

都还在靠实现代码定义真相。

### 2. Firmware 已有逻辑，缺的是仲裁资产

仓库已经实现：
- local file loader
- remote advisory fetch + cache
- parser + candidate resolution
- entity consumer

缺的是：
- authority 层的正式写法
- local vs remote trust semantics 的 contract proof
- repo bundled JSON 的正式 validation / drift guard

### 3. Share Worker 行为测试强于 contract formalization

`share_client` 已有大量错误矩阵测试，但 body 自身仍缺 fixture/snapshot truth。正确方向不是重复测试 retry，而是：
- 固定 generated payload family
- 锁动态字段 normalization
- 补齐 authority source + body snapshot

### 4. Diagnostics endpoints 要分“已冻结”与“未冻结”两层

- `get_city / query_user_cloud`：已由 Phase 1 冻结，直接复用
- `query_command_result / sensor history`：仍偏行为验证，应在 2.6 明确升级路径

### 5. External boundary formalization 必须有 inventory 文档

没有 inventory，就无法让 Phase 3 具体引用“哪条边界由谁拥有、用哪个 fixture、由哪个 drift rule 守护”。

## What Not To Do

1. **不要把 implementation 文件写成 authority source**
   - `firmware_manifest.py` / `report_builder.py` / `helpers.py` 都只是实现，不应兼任真源。

2. **不要把所有 external assets 继续塞进 `tests/fixtures/api_contracts/`**
   - 那个目录是 Phase 1 protocol baseline，不应膨胀成混合垃圾仓。

3. **不要让 remote firmware advisory 决定 certified 真相**
   - 这会直接破坏 firmware trust hierarchy。

4. **不要把 support payload 当成 control plane 私有输出**
   - 它是 external boundary，必须先 formalize 再交给 Phase 3 消费。

5. **不要把 live capture 缺席当借口，继续不落 fixture**
   - 即使没有线上抓包，也可以先为 bundled artifacts、generated payloads、当前已知远端形态落 seed fixtures。

## Common Pitfalls

- 只新增文档，不新增 authority/fixture/drift assets
- 只锁 share/firmware，不把 generated payload family formalize
- 只给 remote endpoint 写 fixtures，不给 bundled JSON / repo artifact 写守护测试
- 把 `get_city / query_user_cloud` 再复制一套新 fixture，制造双真源
- 把 `maintenance` 当成 2.6 的主角，挤占真正高漂移边界的预算

## Recommendation

### Locked Recommendation

**Phase 2.6 的最优执行顺序应为：**

1. 先建立 inventory + authority + boundary hierarchy
2. 再为 share worker / firmware advisory / support payload / diagnostics endpoints 补 contracts + fixtures
3. 最后收口 drift-detection、authority audits、Phase 3 handoff 与 validation closeout

### What Makes This “Advanced”

不是多写几页文档，而是：
- 每条 external boundary 都有 formal owner
- bundled / remote / generated / implementation 的层级真相被锁死
- 既有强协议基线被复用，而不是被重复定义
- Phase 3 得到的是真正可消费的 external-boundary outputs，而不是模糊口头约定

## Confidence

- **High**: `generated` 缺位是当前最关键裂缝，必须在 2.6 正式补齐。
- **High**: firmware 必须拆成 `local trust root / remote advisory / implementation adapters` 三层。
- **High**: `get_city / query_user_cloud` 应复用 Phase 1 fixtures，而不是重新建双真源。
- **Medium**: diagnostics capability family 内部的 fixture 粒度，可在执行时按 drift risk 微调。
- **Medium**: server-side schema 真相仍需外部仓或部署面确认，但不影响本仓先 formalize 客户端边界。
