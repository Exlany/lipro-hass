# Phase 02.6 Architecture: External Boundary Formalization Design

## Objective

把当前分散在协议 helper、share client、firmware loader、diagnostics helper 与 bundled artifacts 中的 external boundary 规则，重建为一套**可审计、可验证、可交接**的正式外部边界体系：

- 每条 external boundary 都有 formal owner
- `docs / fixtures / generated / implementation` 的 authority 与同步方向明确
- firmware 的 trust semantics 不再埋在实现细节里
- generated payload family 拥有正式 contract 与 drift rules
- Phase 3 可以直接消费 Phase 2.6 outputs，而不是继续建立在隐式约定上

## Current Structural Problem

当前外部边界的真实问题不是“功能缺失”，而是**真源分裂**：

- `firmware_support_manifest.json`、remote advisory URLs、parser、candidate resolution、entity consumer 共同决定 firmware 行为，但 authority hierarchy 没有正式写清
- anonymous share / developer feedback / diagnostics support payload 的生成逻辑主要活在 implementation builders 内部，`generated` 没有 authority source
- Phase 1 已冻结 `get_city / query_user_cloud`，但 Phase 2.6 之前没有任何 inventory 把它们与 support / external-boundary story 关联起来
- tests 大量存在，但 fixtures / authority / drift detection 仍未按 external-boundary family 收口

## Target Boundary Topology

### 1. `ExternalBoundaryCatalog`

这是 Phase 2.6 的治理根，而不是新的运行时代码对象。

**Owns**:
- boundary inventory
- owner / authority / fixture / generated / consumer / drift-rule 映射
- external-boundary family 的统一命名与 phase handoff

**Primary artifact**:
- `02.6-BOUNDARY-INVENTORY.md`

### 2. `ShareWorkerBoundary`

**Family scope**:
- anonymous share upload
- install token refresh
- developer feedback submit
- lite payload fallback

**Authority model**:
- payload contract authority：Phase 2.6 fixtures + generated normalization rules
- implementation adapters：`report_builder.py`、`share_client.py`、`manager.py`
- HA-facing control/service owner：`services/share.py`、`services/diagnostics/helpers.py`

### 3. `FirmwareAdvisoryBoundary`

**Family scope**:
- bundled firmware support manifest
- remote firmware advisory endpoints
- OTA advisory rows v1/v2/controller

**Authority hierarchy**:
- **local trust root**：`custom_components/lipro/firmware_support_manifest.json`
- **remote advisory source**：`firmware_manifest.py` 中的 `firmware-support` endpoints
- **implementation adapters**：`firmware_manifest.py`、`core/ota/manifest.py`、`core/ota/candidate.py`
- **runtime consumer**：`entities/firmware_update.py`

**Locked rule**:
- remote advisory 不能单独放宽或定义 `certified`

### 4. `SupportPayloadBoundary`

**Family scope**:
- developer feedback payload
- diagnostics support payload
- developer report / debug snapshot 中的外发 payload

**Authority model**:
- generated payload contract 必须通过 fixture/snapshot + normalization rules formalize
- dynamic fields（如 `generated_at`）必须采用 canonical placeholder / normalization，而不是直接把瞬时值冻结进权威样本

### 5. `DiagnosticsCapabilityBoundary`

**Family scope**:
- `get_city`
- `query_user_cloud`
- `query_command_result`
- sensor history family

**Authority model**:
- `get_city / query_user_cloud`：继续引用 Phase 1 fixtures 作为上游真源
- 其余 diagnostics capability endpoints：在 Phase 2.6 补 inventory、fixture families 与 drift rules

## Authority Model

### Canonical Authority Layers

1. **North-star / planning authority**
   - `docs/NORTH_STAR_TARGET_ARCHITECTURE.md`
   - `.planning/baseline/AUTHORITY_MATRIX.md`
   - `.planning/baseline/VERIFICATION_MATRIX.md`

2. **Boundary-family authority**
   - `02.6-BOUNDARY-INVENTORY.md`
   - per-family fixture families
   - per-family owning tests / drift audits

3. **Implementation adapters**
   - `custom_components/lipro/**`
   - 只能消费 boundary authority，不得反向定义 external truth

### Generated Payload Rule

对 generated payload family，必须定义：
- which builder owns shape assembly
- which fields are deterministic
- which dynamic fields are normalized
- which redaction rules apply before fixture comparison
- which tests are the owning drift guard

## Canonical Boundary Inventory

Phase 2.6 至少应覆盖以下 boundary families：

| Boundary | Target Family | Current Authority | Required Upgrade |
|---|---|---|---|
| anonymous share upload | `ShareWorkerBoundary` | implementation-heavy | 增加 payload fixtures + authority row + drift audit |
| developer feedback payload | `SupportPayloadBoundary` | implementation-heavy | 增加 generated contract + body snapshot + authority row |
| firmware local manifest | `FirmwareAdvisoryBoundary` | bundled artifact but undocumented | 正式写成 local trust root + repo validation |
| firmware remote advisory | `FirmwareAdvisoryBoundary` | implementation-heavy | 明确 advisory-only + remote fixtures |
| OTA advisory rows | `FirmwareAdvisoryBoundary` | behavior tests | 增加 row fixtures + trust semantics proof |
| `get_city / query_user_cloud` | `DiagnosticsCapabilityBoundary` | Phase 1 fixture truth | 登记到 inventory，禁止重复定义 |
| `query_command_result / sensor history` | `DiagnosticsCapabilityBoundary` | behavior tests | 增加 boundary fixtures / stronger drift rule |
| HA diagnostics export | `SupportPayloadBoundary` | support tests | 明确 export contract 与 external payload关系 |

## Cross-Phase Guardrails

- **Phase 1 truth 不在本 phase 重写**：`get_city / query_user_cloud` 的 protocol contract 继续以 Phase 1 为准。
- **Phase 2.5 root ownership 不在本 phase 改写**：external boundaries 必须面向 unified protocol root 叙事，而不是复活 split-root semantics。
- **Phase 3 只消费本 phase outputs**：`DiagnosticsSurface / SystemHealthSurface` 不得再自定义 authority source。
- **Phase 5 runtime ownership 不在本 phase 提前吞并**：外部边界 formalization 不等于 runtime public surface formalization。
- **Server truth 不在本仓拍板**：若边界真实 schema 在外部服务端，本 phase 只 formalize 客户端可知 contract 与本地 drift guards。
