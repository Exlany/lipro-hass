<domain>
## Phase Boundary

本阶段的边界不是“再补几条测试”，而是：

**把 `share / firmware advisory / support payload / external endpoints` 从散落实现与行为约定，收敛为具备 formal owner、authority source、fixture、drift detection 的正式 external-boundary plane。**

本阶段必须完成：
- 建立一份明确 external boundary inventory，至少覆盖：`anonymous share upload`、`developer feedback/support payload`、`firmware advisory`、`diagnostics capability endpoints`
- 为每条边界写清：`owner / authority source / fixture path / generated artifact / primary consumers / drift risk / verification proof`
- 定义 `docs / fixtures / generated / implementation` 的同步方向，补上当前缺失的 `generated` 真源口径
- 把 firmware 的 `local trust root` 与 `remote advisory source` 严格拆开，禁止 remote 数据单独决定 `certified`
- 复用 `Phase 1` 已冻结的 `get_city / query_user_cloud` 契约真相，明确它们在 Phase 2.6 只是被纳入 external-boundary inventory，而不是重新定义协议合同
- 为 Phase 3 提供可引用的 external-boundary outputs，避免 control plane 继续建立在隐式约定之上

本阶段明确**不做**：
- 重新设计 protocol root（属于 Phase 2 / 2.5）
- 重新设计 control plane / domain plane / runtime plane 正式边界（属于 Phase 3 / 4 / 5）
- 改写 `lipro-share-worker` 服务端 schema 真相；本仓库只能 formalize 客户端约束与本地 fixtures
- 线上真实 payload 抓取与回灌；live capture 只能作为后续补充输入
- 固件二进制分发链路治理；`mlink.mzres.com` 二进制分发不属于本集成的正式 owner 范围
</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- `custom_components/lipro/firmware_support_manifest.json` 是 firmware certification 的**本地 trust root**，不是普通样例数据
- `custom_components/lipro/firmware_manifest.py` 中的远端 `firmware-support` URL 只属于 **remote advisory source**，不得单独定义 `certified`
- `custom_components/lipro/core/ota/manifest.py`、`firmware_manifest.py`、`entities/firmware_update.py` 属于 firmware boundary 的 implementation adapters / consumers，不是 authority source
- `anonymous share` 与 `developer feedback` 的 `generated payload` 必须拥有显式 contract owner、动态字段归一化规则与 fixture/snapshot 真源；`report_builder.py` / `helpers.py` 不能继续默默兼任真源
- `get_city / query_user_cloud` 的 canonical contract 真相继续归属 `Phase 1` fixture baseline；Phase 2.6 只负责把它们纳入 external-boundary inventory 与 authority rules
- 每条 external boundary 至少必须交付：`1 owner + 1 authority rule + 1 fixture family + 1 drift rule + 1 owning test surface`
- `generated` 必须在 `AUTHORITY_MATRIX.md` 中拥有显式 authority 行；否则 `INTG-02` 不成立
- implementation 代码永远只是消费 authority / fixtures，不得反向定义外部边界真相

### Claude's Discretion
- external-boundary inventory 是落成单独文档，还是 phase architecture 附表，只要能被 Phase 3 / reviews / baseline 明确引用即可
- fixture families 的目录命名（如 `tests/fixtures/external_boundaries/share_worker/`）可按执行时的测试组织微调
- diagnostics capability endpoints 的分组粒度，可按 `support-only` / `vendor endpoint` 二级视图组织，只要 authority 与 drift rules 清晰即可
</decisions>
