# Phase 7.1 Research

**Phase:** `7.1 Protocol Boundary Schema/Decoder 收口`
**Source milestone research:** `.planning/research/SUMMARY.md`
**Date:** 2026-03-13

## Findings

- 最优做法是先建立 `decoder family + schema registry + decode result`，再决定局部 schema backend。
- v1.1 初期不要求全域引入重型 schema 库；优先建立边界与 authority。
- REST/MQTT 都必须复用同一 boundary truth，不允许形成 `REST decoder truth` / `MQTT decoder truth` 两套变体。
- replay harness 与 exporter 都会依赖 decode result；因此 Phase 7.1 需要顺手定义稳定观察点。

## Required Outputs

- `core/protocol/boundary/` target design
- boundary inventory / decoder family / schema registry contract
- REST/MQTT integration points and migration sequence
- authority + fixture + drift-detection handoff for Phase 7.4 replay work

## Risks To Avoid

- decoder 成为新 root
- raw payload leakage
- schema backend 选型先行、边界设计滞后
- authority docs / fixtures / implementation 三口径漂移
