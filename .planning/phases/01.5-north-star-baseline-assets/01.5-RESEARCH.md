# Phase 01.5 Research: North-Star Baseline Assets

**Updated:** 2026-03-12
**Research mode:** baseline formalization
**Decision stance:** 先把正式基准资产化，再允许大规模结构迁移

## Research Framing

本 phase 的核心问题不是“文档够不够多”，而是：
- 哪些 baseline docs 必须成为后续 phase 的正式输入真源？
- 最小可运行 guard 应该在哪一层落下，才能在不引入重框架的前提下阻止明显回退？
- 如何把 requirements、docs、tests、phase handoff 串成统一验证闭环？

## Existing Assets Worth Formalizing

### 1. Target Topology
- `.planning/baseline/TARGET_TOPOLOGY.md` 已定义五平面 formal roots、关键组件与目录意图
- 这是 `BASE-01` 与 `ARCH-04` 的核心输入，不应只是阅读材料

### 2. Dependency Matrix
- `.planning/baseline/DEPENDENCY_MATRIX.md` 已定义 allowed / forbidden dependencies 与 guard seeds
- 这是 `BASE-02` 与 `ASSR-06` 的直接落点

### 3. Public Surfaces
- `.planning/baseline/PUBLIC_SURFACES.md` 已定义 canonical / transitional / forbidden surfaces
- 后续 phase 若不引用它，很容易重新把 compat shell 提升回正式 root

### 4. Verification Matrix
- `.planning/baseline/VERIFICATION_MATRIX.md` 已定义 requirement → artifact → test → phase 的映射
- 这是 `BASE-03` 的正式落点，也是 phase 验收的唯一统一入口

### 5. Authority Matrix
- `.planning/baseline/AUTHORITY_MATRIX.md` 已定义 docs / fixtures / generated / implementation 的同步方向
- 这是 `2.6` 以及后续 docs hygiene 的基础

## Recommended Guard Strategy

### Minimal, Static, Local
优先在 `tests/meta/` 下建立最小静态 guards：
- dependency-direction seeds
- public-surface seeds
- baseline asset presence / alignment seeds

理由：
- 轻量、可读、可在重构前就跑起来
- 不需要引入额外框架
- 与现有 `tests/meta/test_install_sh_guards.py` 风格一致

## Do Not Overbuild

- 不要在 `1.5` 就试图完成 full architecture enforcement
- 不要引入通用规则引擎或复杂 AST 框架
- 不要让 baseline docs 变成另一套 roadmap；它们是执行输入，不是第二份项目章程

## Recommendation

1. `01.5-01` 先锁 baseline asset pack 的内容、边界与同步规则
2. `01.5-02` 再用 `tests/meta/` 落最小 dependency/public-surface guards
3. `01.5-03` 最后把 verification matrix 与 phase acceptance/handoff 串成正式闭环

## Confidence

- **High**: baseline assets 作为大规模重构前的正式输入，符合本项目“目标态先行、验证先行”的主线
- **High**: `tests/meta/` 作为 seed guards 落点，符合现有仓内风格且成本低
- **Medium**: seed guards 的覆盖边界需要执行期结合目录结构微调
