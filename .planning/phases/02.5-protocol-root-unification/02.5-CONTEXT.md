<domain>
## Phase Boundary

本阶段的职责不是“给 REST façade 再包一层名字”，而是**完成 protocol plane 的正式收口**：

- 把 `REST / MQTT` 从两个并存的实现重心，统一到单一正式协议根：`LiproProtocolFacade`
- 把 `auth session / refresh recovery / telemetry / diagnostics context / canonical normalized contracts` 上收为 protocol-root owned truth
- 把 **payload normalization** 明确裁定为 protocol plane 的边界职责，满足 `PROT-05`
- 让 `LiproRestFacade`、`LiproMqttFacade` 退回到 child façade 角色，满足 `PROT-06`
- 把 `LiproClient`、legacy wrapper、历史 split-root 命名降级为显式 compat shell，只能跟随，不能继续定义正式入口

本阶段**不做**：
- 重新设计 external-boundary authority（属于 Phase 2.6）
- 重新设计 control plane 正式组件（属于 Phase 3）
- 重新定义 runtime / domain plane 的终态 public surfaces（属于 Phase 4 / 5）
- 让 MQTT 继续以“实时特例”绕开统一协议根
</domain>

<decisions>
## Implementation Decisions

### Locked Decisions
- `LiproProtocolFacade` 是唯一正式 protocol-plane root
- `LiproRestFacade` / `LiproMqttFacade` 是 child façades，不再承认“REST 根 + MQTT 根”双模型
- protocol plane 必须对上游供应商 payload 做 canonical normalization；runtime / control / domain 不直接消费 raw vendor shape
- 共享 truth（auth、telemetry、protocol diagnostics context、canonical protocol contracts）优先向上收敛，而不是向下复制
- `LiproClient` 不属于终态正式设计；若阶段性保留，只能作为显式 compat shell
- 新架构不承认 `mixin` 作为正式设计模式

### Claude's Discretion
- 统一协议根最终落在 `core/api/` 过渡层还是新建 `core/protocol/` 逻辑包，只要 public surface 唯一且 child façade 边界清晰即可
- root-owned shared collaborators 的精确模块切分可按实现可读性裁决
- compat shell 的拆除顺序可按 direct-consumer 迁移波次决定，但必须有 kill condition 与 residual owner
</decisions>

<downstream_contract>
## Downstream Contract

### Contract for Phase 2.6
`Phase 2.6` 只能建立在统一协议根之上：
- external-boundary owner 必须附着到统一 protocol root 或其 root-owned canonical contracts
- `share / firmware / support payload / diagnostics capability` 不得再绑定到 transport-specific root semantics
- authority / fixture / generated artifact 的同步方向必须引用 `Phase 2.5` 收口后的正式 public surface

### Contract for Phase 3
`Phase 3` 不直接依赖 protocol root；它应继续通过 runtime public surface 与 control contracts 协作。

`Phase 2.5` 对 `Phase 3` 的作用是提供：
- 已收口的 unified protocol root 前提
- 已消灭的 split-root public semantics
- 可被 runtime plane 安全消费的 canonical contracts

`Phase 3` 不得重新感知：
- child façade 内部 transport helpers
- split-root compat shell
- 原始供应商 payload 形态

<specifics>
## Specific Ideas

- `LiproProtocolFacade` 拥有 shared auth/session、request policy、protocol telemetry、protocol diagnostics context、canonical contracts registry
- `LiproRestFacade` 聚焦 REST / IoT 请求执行、endpoint collaborators、payload normalizers
- `LiproMqttFacade` 聚焦 MQTT connection / subscription / ingress，但是其 diagnostics / telemetry / normalized outputs 要向 root 对齐
- runtime-facing allowed consumers 最终只能把 protocol plane 当作**单一正式 public surface** 处理
</specifics>
