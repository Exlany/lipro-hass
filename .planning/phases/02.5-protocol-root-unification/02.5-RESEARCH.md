# Phase 02.5 Research: Protocol Root Unification

**Updated:** 2026-03-12
**Research mode:** architecture consolidation
**Decision stance:** 脱离历史成本约束，以统一协议平面根为唯一终态判断标准

## Research Framing

本 research 关注的问题不是“如何把 MQTT 接上现有结构”，而是：

- 在最优终态里，protocol plane 的唯一正式根应该怎样定义？
- REST / MQTT 的 shared truth 应该放在什么层级拥有？
- 哪些能力必须上收至 protocol root，哪些保留在 child facades 即可？

## Standard Stack

### 1. Home Assistant Runtime Ownership

**Use**:
- `ConfigEntry.runtime_data` 作为 integration runtime root 的承载点
- high-level integration wiring 面向稳定 façade / root，而不是面向多个协议分裂入口

**Why**:
- 这更符合 HA strict typing 与 runtime ownership 的方向，也让 unload / diagnostics 更容易统一。

### 2. Python Explicit Boundary Modeling

**Use**:
- `dataclass(slots=True)` 承载 shared session / telemetry / diagnostics context
- `TypedDict` 承载 canonical protocol contracts
- `Protocol` 仅用于 root 与 child-facade 协作接口的静态约束

### 3. Transport-Specific Infrastructure

**Use**:
- REST 保持 `aiohttp.ClientSession` + explicit executor
- MQTT 保持 dedicated connection/subscription runtime helpers
- transport-specific implementation can stay separate, but root ownership must be singular

## Architecture Patterns

### 1. Unified Root + Child Facades

**Use**:
- `LiproProtocolFacade` as the only formal protocol-plane root
- `LiproRestFacade` and `LiproMqttFacade` as child façades

**Why**:
- 这样既保留 transport-specific implementation clarity，又不会让协议平面重新分裂成两个正式根。

### 2. Shared Truth Rises Upward

**Use**:
- auth session, telemetry schema, diagnostics context, canonical protocol contracts belong at the root boundary first

**Why**:
- shared truth 如果分散在 REST / MQTT 两侧，最终会导致 duplicated drift and inconsistent support surfaces。

### 3. Consumer Simplicity Rule

**Use**:
- runtime / control plane should depend on one protocol root contract
- child-facade details should remain internal to the protocol plane

**Why**:
- 上层不应该知道“我是走 REST 还是 MQTT 根”；那是 protocol plane 的内部组织问题。

## Don't Hand-Roll

- 不要再发明第三套“协议协调层”与 façade 并行存在
- 不要让 MQTT 因为“实时”就跳过统一协议根
- 不要把 observability 做成各 transport 各一份不兼容指标
- 不要用 compat shell 长期保留 split-root semantics

## Common Pitfalls

1. 只统一命名，不统一 ownership
2. 只挂一个空 root，实际 shared truth 仍散落在 REST / MQTT 两边
3. 把 root 做成 service locator，丢失明确边界
4. 让 diagnostics / telemetry 继续沿 transport 分裂输出

## Code Examples

### Example A: Unified Root

```python
@dataclass(slots=True)
class ProtocolSessionState:
    access_token: str | None = None
    refresh_token: str | None = None

class LiproProtocolFacade:
    def __init__(self, rest: LiproRestFacade, mqtt: LiproMqttFacade) -> None:
        self.rest = rest
        self.mqtt = mqtt
```

### Example B: Root-Owned Telemetry

```python
@dataclass(slots=True)
class ProtocolTelemetry:
    request_count: int = 0
    mqtt_reconnects: int = 0
```

## Recommendation

1. Phase 2 结束后，不要把 `LiproRestFacade` 当成终点；它只是统一协议根的前置台阶
2. `Phase 2.5` 必须把 shared auth / telemetry / diagnostics / protocol contracts 向 `LiproProtocolFacade` 上收
3. MQTT 子面必须被视为 protocol plane 的正式子门面，而不是 runtime plane 的旁路设施
4. Phase 3 及后续 phases 应只依赖统一协议根的正式 public contract

## Confidence

- **High**: 单一正式协议根优于 split-root semantics，与北极星“单一正式主链”一致
- **High**: shared truth 上收至 root 比分别在 REST / MQTT 两侧维护更稳健
- **Medium**: root 与 child-facade 的精确文件落点，需要结合 Phase 2 实际改造成果微调
