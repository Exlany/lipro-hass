# Phase 02.5 Architecture: Unified Protocol Root

## Objective

完成 protocol plane 的正式终态收口：
- 让 `LiproProtocolFacade` 成为唯一正式协议根
- 让 `LiproRestFacade` / `LiproMqttFacade` 成为明确 child façades
- 让 canonical normalized contracts 在 protocol boundary 完成，而不是继续散落在 transport-specific internals

## Requirement Alignment

| Requirement | What This Phase Must Decide | Expected Output |
|---|---|---|
| `PROT-05` | payload normalization 的正式 owner 必须在 protocol plane | root-owned contract boundary + tests + handoff proof |
| `PROT-06` | `REST / MQTT` 必须统一收口到单一 `LiproProtocolFacade` | unified root design + consumer migration + compat cleanup |

## Formal Ownership Rules

| Concern | Formal Owner | Must Not Be Owned By |
|---|---|---|
| protocol public entry | `LiproProtocolFacade` | `LiproRestFacade`, `LiproMqttFacade`, `LiproClient`, `LiproMqttClient` |
| canonical normalized contracts | protocol root boundary | runtime plane, control plane, entity/platform adapters |
| auth session / refresh recovery | protocol root | per-transport duplicate helpers |
| telemetry / protocol diagnostics context | protocol root | split REST/MQTT sidecars, `DiagnosticsSurface` |
| REST transport execution | `LiproRestFacade` | runtime/control consumers |
| MQTT connection / subscription / ingress | `LiproMqttFacade` | runtime/control consumers |
| support payload authority / HA diagnostics export | Phase 2.6 / Phase 3 surfaces | protocol-root internals |
| compatibility naming | explicit compat adapters only | any formal public surface |

## Boundary Rules

### 1. Single Public Surface Rule
所有**被允许跨平面依赖 protocol plane 的消费者**（主要是 runtime plane 与显式登记的 compat adapters）只允许面向 `LiproProtocolFacade`，不允许再把 child façade 当作正式根。

### 2. Normalization-at-Boundary Rule
raw vendor payload 进入 transport adapter 后，必须在 protocol plane 完成 canonical normalization；
runtime plane 与其他获准读取 canonical contracts 的协作者只消费归一化后的协议 contracts。

### 3. Child-Facade Rule
`LiproRestFacade`、`LiproMqttFacade` 保留 transport-specific implementation clarity，但不再拥有 protocol-wide truth。

### 4. Diagnostics-Scope Rule
`ProtocolDiagnosticsContext` 只表示 protocol-owned telemetry / diagnostics context；
它**不等于** `DiagnosticsSurface`、`SystemHealthSurface`、support payload authority 或 HA diagnostics export。
这些职责分别留给 `Phase 2.6` 与 `Phase 3` 正式收口。

### 5. Compat Demotion Rule
兼容壳可以短期存在，但必须：
- 显式标识 residual owner
- 有 direct-consumer 清单
- 有明确 kill condition
- 不得继续定义 public direction

## Selected Execution Stance

### 1. 逻辑归属固定为 `core/protocol/`
Phase 2.5 不再把“是否新建 protocol 包”留作开放问题。正式裁决如下：

```text
custom_components/lipro/core/protocol/
├── facade.py
├── session.py
├── telemetry.py
├── diagnostics_context.py
├── contracts.py
└── compat.py
```

现有 `core/api/` 与 `core/mqtt/` 继续承载 transport-specific child façade 与 helpers；
`core/protocol/` 负责 root-owned shared truth 与唯一正式入口。

### 2. REST child façade 的职责
`LiproRestFacade` 保留：
- signed REST / IoT transport execution
- endpoint collaborators
- request/response codec 与 REST-specific fallback

但它不再定义：
- protocol public direction
- protocol-wide diagnostics context
- protocol-wide canonical contract ownership

### 3. MQTT child façade 的职责
`LiproMqttFacade` 保留：
- broker connection / reconnect loop
- subscription management
- message ingress parsing 与 MQTT transport-specific state

但它不再定义：
- runtime-facing public root
- protocol-wide telemetry truth
- canonical contract authority

### 4. Root-owned shared truth
`LiproProtocolFacade` 必须拥有并暴露：
- `ProtocolSessionState`：shared auth/session visibility
- `RequestPolicy`：protocol-wide pacing / retry / backoff policy owner
- `ProtocolTelemetry`：REST/MQTT shared telemetry counters 与 protocol diagnostics hooks
- `ProtocolDiagnosticsContext`：protocol-owned diagnostics context（非 HA diagnostics surface）
- `CanonicalProtocolContracts`：所有对上层可见的 canonical normalized protocol contracts

## Logical Package Direction

终态逻辑结构为：

```text
custom_components/lipro/core/protocol/
├── facade.py
├── session.py
├── telemetry.py
├── diagnostics_context.py
├── contracts.py
├── compat.py
├── rest/
└── mqtt/
```

执行期可在现有 `core/api/` 与 `core/mqtt/` 上分步落地，但**逻辑归属已锁定为统一 protocol root**，不再允许继续沿 split-root 语义演进。

## Downstream Handoff

### Handoff to Phase 2.6
Phase 2.6 只能基于以下前提继续：
- external-boundary contracts 引用 root-owned canonical contracts
- firmware/share/support payload 不再各自绑定旧 split-root 语义
- authority matrix 可明确写出“root -> fixture/generated/implementation”的同步方向
- `LiproProtocolFacade` 已成为唯一正式协议入口，child façade 只作 transport-specific implementation detail

### Handoff to Phase 3
Phase 3 不直接成为 protocol root 的正式 consumer；
它应继续通过 runtime public surface 与 control contracts 协作，只把 unified protocol root 视为**上游已收口前提**，而不是新的 control-plane 直连入口。

`Phase 2.5` 对 `Phase 3` 的作用是提供：
- 已收口的 unified protocol root 前提
- 已消灭的 split-root public semantics
- 可被 runtime plane 安全消费的 canonical contracts

`Phase 3` 不得重新感知：
- child façade 内部 transport helpers
- split-root compat shell
- 原始供应商 payload 形态

## Execution Slices

### Plan 02.5-01
锁定 architecture、ownership、governance、residual accounting。

### Plan 02.5-02
在代码中建立 unified root skeleton，并把 shared auth / telemetry / protocol diagnostics context / normalized contracts 上收到统一协议根。

### Plan 02.5-03
迁移 runtime-facing allowed consumers，清退 split-root public surface 与 compat adapters，并回写 baseline / residual / kill 真源。

## Exit Conditions

只有以下条件同时满足，Phase 2.5 才算真正 closeout：
- `LiproProtocolFacade` 成为唯一正式协议根
- `PROT-05` 的 normalization ownership 在代码与测试中都能证明
- runtime-facing allowed consumers 不再理解 split-root semantics
- `LiproClient` / `LiproMqttClient` / legacy wrapper 只剩显式 compat shell，且已登记 kill path
- `Phase 2.6` 与 `Phase 3` 所需 handoff artifacts 已经存在
