# Requirements: Lipro-HASS North Star Rebuild

**Defined:** 2026-03-12
**Core Value:** 把 `lipro-hass` 收敛为面向 Home Assistant 的、五平面分离、协议边界可审计、运行时单一主链、领域能力单一真源、保障平面正式化的终态架构。

## v1 Requirements

### Baseline Assets

- [x] **BASE-01**: 五平面目标组件、canonical public surfaces、目标目录映射必须形成单一基准清单
- [x] **BASE-02**: allowed / forbidden dependency matrix 必须显式落表，并能映射到 architecture guards
- [x] **BASE-03**: requirement → artifact → test → doc → phase 的 verification matrix 必须存在并作为 phase 验收真源

### Architecture

- [x] **ARCH-01**: 全层只承认显式组合为正式设计，不再接受 `mixin` 聚合作为合法终态模式
- [x] **ARCH-02**: 命令、刷新、状态应用、MQTT 生命周期必须各自只有一条正式主链
- [x] **ARCH-03**: 历史兼容层不得再定义架构真源，只能作为显式迁移残留并拥有删除条件
- [x] **ARCH-04**: 五平面依赖方向必须在大规模重构前就具备基线级护栏，而不是等到收尾阶段才开始约束

### Protocol Boundary

- [x] **PROT-01**: 所有逆向协议不可变约束（固定密钥、签名算法、固定字段、topic/path 约定）必须集中到协议边界管理
- [x] **PROT-02**: 关键外部入口必须具备 golden fixtures 与 contract matrix，足以发现协议漂移
- [x] **PROT-03**: `REST / IoT` 协议入口必须收敛到正式 `LiproRestFacade`，不再依赖历史 mega client / mixin 聚合
- [x] **PROT-04**: `LiproClient` 不得再被视为正式架构根，只能作为短期 compat shell，直至删除
- [x] **PROT-05**: protocol plane 必须在边界完成 payload normalization，运行面与领域面不直接消费原始供应商形态
- [x] **PROT-06**: `REST / MQTT` 终态必须统一收口到单一 `LiproProtocolFacade`
- [x] **PROT-07**: `AuthSession / retry / rate-limit / reauth / request policy / payload normalizers` 必须闭环于 protocol plane，不得泄漏到 runtime/control

### Control Plane

- [x] **CTRL-01**: `config entry`、`reauth`、`options`、`diagnostics`、`system health`、`services` 必须形成正式 control plane 叙事
- [x] **CTRL-02**: control plane 与 runtime plane 之间必须通过稳定 public surface 对接，而不是散落 coordinator backdoor
- [x] **CTRL-03**: 控制面变化必须具备 lifecycle、support surface 与 redaction 级别的测试覆盖
- [x] **CTRL-04**: `EntryLifecycleController / ServiceRegistry / DiagnosticsSurface / SystemHealthSurface` 必须成为明确的目标组件与正式交付物

### Domain Plane

- [x] **DOM-01**: 设备能力必须存在单一 source-of-truth，平台、实体、命令、属性描述围绕同一 capability model 演进
- [x] **DOM-02**: 平台层只能消费 capability snapshot / projection，不得持有第二套规则系统
- [x] **DOM-03**: 命令对象、属性描述与能力判断必须共享统一的领域契约
- [x] **DOM-04**: 新设备或新能力接入时，新增点应主要落在 capability model，而不是扩散到多个平台分支
- [x] **DOM-05**: `CapabilityRegistry + CapabilitySnapshot` 必须成为命名化终态交付物，平台/实体只能消费 snapshot/projection

### Runtime Plane

- [x] **RUN-01**: `Coordinator` 必须保持 runtime plane 唯一正式编排根
- [x] **RUN-02**: `RuntimeOrchestrator` 与 `RuntimeContext` 必须是显式 wiring / callback 注入边界
- [x] **RUN-03**: MQTT 恢复、命令确认、轮询刷新、状态应用等关键路径必须通过 formal runtime/protocol public surface 协作
- [x] **RUN-04**: runtime invariants 必须具备自动化验证，能防止旁路刷新、旁路写状态、重复订阅、异常吞没等回归

### Assurance Plane

- [x] **ASSR-01**: contract tests 必须成为协议重构的正式回归防线，而不是样例快照附属品
- [x] **ASSR-02**: 关键运行信号必须可观测，包括 auth recovery、MQTT reconnect、command confirmation、refresh latency
- [x] **ASSR-03**: 必须具备 architecture enforcement 或 meta guard，阻止跨层直连、compat 回流、双主链复发
- [x] **ASSR-04**: 测试代码必须随正式架构同步现代化，不能继续围绕旧 public names 或旧路径打补丁
- [x] **ASSR-05**: CI 质量门必须能证明“结构未退化”，不仅证明“功能仍可跑”
- [x] **ASSR-06**: dependency-direction guards 与 public-surface guards 必须在大规模结构迁移前就具备最小可运行版本

### Integration Boundaries

- [x] **INTG-01**: anonymous share、firmware advisory、support payload、external endpoint 必须具备 owner、contract、fixture 与 drift detection
- [x] **INTG-02**: `docs / fixtures / generated / implementation` 必须定义 authority source 与同步方向，避免外部边界多口径漂移

### Governance

- [x] **GOV-01**: 全部 `404` 个 Python 文件都必须纳入治理矩阵，归类为 `保留 / 重构 / 迁移适配 / 删除`
- [x] **GOV-02**: 所有 compat adapter、旧命名、影子模块、过时文档都必须记录 owner、phase 与删除触发条件
- [x] **GOV-03**: 北极星文档、开发文档、`.planning/`、测试矩阵、代码状态必须保持单一口径与同步更新
- [x] **GOV-04**: 阶段完成时必须留下可验证的收尾状态：残留台账、删改列表、验收命令、未完成事项说明
- [x] **GOV-05**: `FILE_MATRIX` 必须提升为 file-level 视图，包含 owner phase、residual link、delete gate、完成百分比

## v2 Requirements

### Evolution

- **EVOL-01**: 若协议边界复杂度继续显著上升，可在 boundary plane 评估更强 schema/decoder 方案（如 `msgspec` 或 `pydantic v2` 的局部引入）
- **EVOL-02**: 若架构回归风险继续升高，可引入更强的 import / dependency rule 工具提升 enforcement 强度
- **EVOL-03**: 若 runtime telemetry 需求显著增加，可演进为更正式的 metrics/exporter 体系
- **EVOL-04**: 若协议仿真需求增加，可建立 vendor payload replay / simulator harness

## Out of Scope

| Feature | Reason |
|---------|--------|
| 通用插件框架化 | 背离 HA 插件真实边界，增加抽象噪音 |
| 全局事件总线 | 会削弱单一正式主链与可追踪性 |
| 通用 DI 容器 | 当前规模收益不足，显式装配更清晰 |
| 全域重型 schema 框架 | 运行时与样板成本过高，当前收益不足 |
| 与协议/架构收敛无关的大规模换栈 | 不能直接改善项目核心问题 |
