## Deep Review Findings

### Architecture strengths
- `protocol / control / runtime / governance` 的正式 home 边界继续收紧，没有为了“测试方便”重新把 helper 或 compatibility shell 拉回 public surface。
- `runtime_access_support.py` 的整改方式是对的：没有牺牲 patch seam 去换取表面瘦身，而是在 formal import home 内保住可测性与预算约束。
- 历史治理 guard 本轮从“读取 live docs 验证历史事实”纠偏为“读取 archived snapshots 验证历史事实”，这符合 open-source governance 的长期可维护原则。

### Open-source readiness strengths
- release/public wording 与 developer/runbook wording 已分层：`CHANGELOG.md` / `services.yaml` 不再泄露内部 planning phase 术语，而 maintainers 仍能从开发文档获得足够细节。
- 目录结构虽大，但当前 formal home、support seam、archive truth、execution trace 的边界已经比多数同类 HA 私有集成项目更清晰。
- focused meta guards 继续作为真实治理裁决层，而不仅是字符串快照堆砌。

### Non-blocking observations
- milestone governance 资产仍然偏多，贡献者认知负担主要来自 `.planning` 派生文档数量，而不再是代码 formal-home 混乱；后续若继续优化，最值得投入的是进一步降低人工维护的派生文档密度。
- `gsd-tools` 在 nested worktree 的 root detection 仍不稳定，这已被明确降级为非 authority signal；问题不阻塞 `v1.43` closeout，但值得在后续工具链层处理。
